# AI辅助编程与工程化实训 - 个人总结报告

## 一、项目概述

本次实训我独立完成了一个名为 **Wordmemory** 的AI智能背单词全栈Web应用。项目采用前后端分离架构，前端基于 React 18 + TypeScript + Vite 6，后端采用 Python Flask 3 + SQLite，支持用户认证、多种学习模式、词库管理、学习统计等完整功能。

## 二、遇到的最大困难及解决过程

### 1. 数据库锁定问题（最严重）

**问题描述**：后端测试运行时频繁出现 `sqlite3.OperationalError: database is locked` 错误，导致大部分测试失败。

**原因分析**：
- 测试框架并发执行时，多个测试用例同时访问同一个 SQLite 文件数据库
- 每次请求都创建新的数据库连接，且部分请求中调用了 `conn.close()`
- 测试之间没有正确的数据库隔离

**解决过程**：
1. 修改 `DB_PATH` 为环境变量可配置，测试时设置为 `:memory:` 内存数据库
2. 实现单例模式的 `get_sqlite_conn()` 函数，内存数据库模式下复用同一个连接
3. 删除所有 `conn.close()` 调用，避免测试期间关闭内存数据库连接
4. 创建 `close_sqlite_conn()` 辅助函数，仅在生产环境（文件数据库）中关闭连接

### 2. 数据同步与用户隔离问题

**问题描述**：用户登录后，学习记录、统计数据等没有按用户ID正确隔离，导致不同用户看到相同的数据。

**解决过程**：
1. 在所有数据库表中添加 `user_id` 字段
2. 修改所有API接口，添加 `user_id` 过滤逻辑
3. 实现 `get_current_user_id()` 函数从 JWT token 中提取用户ID
4. 退出登录时调用 `resetStore()` 清空本地状态

### 3. 统计数据假数据问题

**问题描述**：前端统计页面展示的学习数据是硬编码的 mock 数据，后端从未写入真实统计数据。

**解决过程**：
1. 在 `create_learning_record()` 函数中添加统计写入逻辑
2. 新建单词时更新 `words_learned` 计数
3. 复习单词时更新 `words_reviewed` 计数
4. 使用 `INSERT OR IGNORE` + `UPDATE` 实现 upsert 操作

### 4. API 字段命名不一致

**问题描述**：后端返回 `snake_case`（如 `words_learned`），前端类型期望 `camelCase`（如 `wordsLearned`）。

**解决过程**：
1. 创建 `snake_to_camel()` 转换函数
2. 在统计数据接口中统一转换为 camelCase
3. 保持学习记录接口已有的手动映射

## 三、对AI辅助编程的心得体会

### 1. AI 是高效的代码生成器

在项目初始化阶段，AI 帮我快速搭建了完整的项目结构，包括：
- 前端组件（页面、状态管理、API封装）
- 后端 API（路由、数据库操作）
- 配置文件（Vite、TailwindCSS、TypeScript）

这节省了大量重复的脚手架工作，让我能专注于业务逻辑。

### 2. AI 需要精确的 Prompt

通过这次实训，我深刻体会到 Prompt 工程的重要性：
- 明确的技术栈要求能避免 AI 使用不适合的技术
- 详细的功能描述能生成更准确的代码
- 标注需要修改的文件能控制代码生成范围

### 3. AI 生成的代码需要严格审查

AI 生成的代码并非完美，存在以下问题需要人工修复：
- 安全漏洞（如 JWT 密钥硬编码、缺少鉴权）
- 逻辑错误（如统计数据未写入、字段命名不一致）
- 工程化问题（如缺失依赖、部署配置错误）

### 4. AI 是优秀的调试助手

当遇到问题时，AI 能快速分析错误信息，提供修复方案。例如数据库锁定问题，AI 帮助我定位到连接管理问题，并提供了内存数据库和单例模式的解决方案。
